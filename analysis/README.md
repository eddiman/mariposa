## Quick Reference

### Critical Security Issues (Fix Immediately)

| Issue | Severity | Location | Time to Fix |
|-------|----------|----------|-------------|
| Command Injection | HIGH | `api/src/routes/config.ts:78-111` | 2 hours |
| No Auth | CRITICAL* | All API routes | 8 hours (Phase 2) |
| Open Network Binding | MEDIUM | `api/src/config.ts:6` | 5 minutes |
| CORS Wildcard | LOW | `api/src/index.ts:18` | 30 minutes |
| No Upload Rate Limit | MEDIUM | `api/src/routes/assets.ts` | 1 hour |

*Acceptable for local-only development, critical for any network exposure.

### Architecture Strengths

✅ Local-first storage (no database)  
✅ Non-invasive metadata (sidecar pattern)  
✅ Excellent UI/UX (React Flow + TipTap)  
✅ Strong type safety (TypeScript + Zod)  
✅ Good test coverage (87 tests, ~80%)  
✅ Clean separation of concerns  

### Architecture Weaknesses

❌ Tight filesystem coupling (no abstraction layer)  
❌ No conflict resolution (last-write-wins)  
❌ Image positions in localStorage (should be server-side)  
❌ No undo for destructive operations  
❌ Search is O(N) full scan (slow for large KBs)  
❌ No sidecar integrity checks (corrupted file = lost layout)  

### Integration Strategy

**Phase 1 → 2 Transition Point**:
- Replace `configService` with `registryService` (reads Adjutant's `knowledge_bases/registry.yaml`)
- Add session token auth (generated by Adjutant on startup)
- Enforce read-only KB access levels
- Add `adjutant web` CLI commands

**Key Design Decision**: Mariposa becomes a **capability of Adjutant**, not a standalone app. It discovers KBs via Adjutant's registry and respects Adjutant's access control.
